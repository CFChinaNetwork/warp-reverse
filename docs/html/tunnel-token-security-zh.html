Cloudflared Token 安全部署手册

目标：客户将 CF tunnel token 安全存储在自己的 AWS Secrets Manager，CMI 使用 Docker 容器运行 cloudflared，token 通过 --token-file 注入容器，不以明文出现在任何配置文件或进程参数中。
版本：2026-04  |  基于实际部署环境（Amazon Linux 2 + Docker 25.0.14）验证  |  预计完成时间：90 分钟

每个步骤标注了执行位置：

整体架构与安全边界说明

为什么需要这套方案

传统部署方式（--token eyJ... 直接写进配置）存在以下静态暴露问题：

  • docker inspect <container> → token 明文出现在 Args 字段
  • cat /etc/systemd/system/cloudflared.service → token 明文可见
  • ps aux → token 明文出现在进程参数

本方案的两层隔离

  • 存储层隔离：token 存储在客户自己的 AWS Secrets Manager，CMI 没有客户的 AWS 账号权限，无法访问。
  • 运行层隔离：cloudflared 运行在 Docker 容器内，使用 --token-file 读取密钥文件，token 值不出现在 docker inspect、ps aux 或任何配置文件的明文位置。
┌──────────────────────────────────────────────────────────────────────┐
│            客户自己的 AWS 账号（CMI 无任何访问权限）                    │
│                                                                      │
│   AWS Secrets Manager                                                │
│   └── prod/cloudflare/warp-reverse-tunnel-token-aws                 │
│         token 加密存储（AES-256）                                    │
│         访问控制：只允许绑定到宿主机的 IAM Role 读取                  │
│         CMI 账号无 IAM 权限 → GetSecretValue 返回 403               │
└─────────────────────────────┬────────────────────────────────────────┘
                              │ Instance Profile 自动认证（无需存储密钥）
                              │
┌─────────────────────────────▼────────────────────────────────────────┐
│                       CMI 提供的宿主机（EC2）                          │
│                                                                      │
│  AWS CLI 启动时拉取 token → 写入 /opt/cloudflared/secrets/tunnel.token│
│  目录权限：700（仅 root 可进入）                                      │
│  文件权限：604（root 读写，容器内非 root 用户只读）                    │
│                                                                      │
│  docker-compose.yml（CMI 运维可见，但无 token 值）                    │
│  └── secrets.tunnel_token.file → /opt/cloudflared/secrets/tunnel.token
│                                                                      │
│  cloudflared 容器（UID 65532 运行）                                   │
│  └── command: tunnel run --token-file /run/secrets/tunnel_token      │
│        ↑ 文件路径，不是 token 值                                      │
└─────────────────────────────┬────────────────────────────────────────┘
                              │ QUIC / TLS 1.3（4条连接到 CF 边缘）
                              ▼
                     Cloudflare 全球网络
透明说明：本方案的剩余暴露窗口
任何运行中的进程，其持有的凭证在技术上都可以被宿主机的 root 用户通过内存读取获得。cloudflared 也不例外。具体路径：
sudo docker exec cloudflared-prod cat /run/secrets/tunnel_token → token 明文可见

这是 Linux 操作系统的基本设计，不是 cloudflared 或本方案的缺陷。本方案解决的是静态存储暴露（配置文件、docker inspect、ps aux），而非运行时内存暴露。 对于运行时风险，CMI 通过合规审计和权限管控机制（见文末"合规审计"章节）进行约束。

MAC Step 1：配置 Mac 上的 AWS CLI 认证
目的：让你的 Mac 能通过命令行操作 AWS，用于后续创建 Secret、IAM Policy、Role 等。如果偏好网页操作，所有步骤也可在 AWS Console 完成，本步可跳过。

1-A：确认 AWS CLI 已安装

aws --version
预期输出： 
aws-cli/2.x.x Python/3.x.x Darwin/...
如未安装，访问 https://aws.amazon.com/cli/ 下载 macOS 安装包。

1-B：创建 AWS Access Key

  1. 登录 https://console.aws.amazon.com
  2. 右上角点击用户名 → Security credentials
  3. 向下滚动到 Access keys 区域
  4. 点击 Create access key（如按钮为灰色，说明已有 2 个 Key，需先删除一个旧的）
  5. Use case 选择 Command Line Interface (CLI) → 勾选确认 → Next → Create
  6. 页面显示 Access key ID 和 Secret access key，仅显示一次，立即复制保存
如果 Create access key 按钮是灰色：AWS 每个账号最多 2 个 Access Key。进入 Access keys 列表，删除一个最久未使用的旧 Key 后，按钮即可点击。

CF 控制台 Step 3：在 CF Dashboard 创建 Tunnel 并获取 Token
目的：在 Cloudflare 创建 tunnel，拿到 token 字符串（以 eyJ 开头）。这一步只需复制 token，不执行页面给出的任何安装命令。
  1. 打开 https://one.dash.cloudflare.com，选择你的账号
  2. 左侧菜单：Networks → Connectors → Cloudflare Tunnels
  3. 点击 Create a tunnel
  4. Connector type 选择 Cloudflared → Next
  5. Tunnel name 填写一个有意义的名字，例：warp-reverse-tunnel
  6. 点击 Save tunnel
  7. 下一页，Operating system 选择 Docker
  8. 页面显示命令如下，只复制 eyJ 开头的 token 字符串，不执行整条命令：
页面显示：
docker run cloudflare/cloudflared:latest tunnel --no-autoupdate run --token eyJhIjoiMDI5N2Rm...（这里是完整 token）
                                                                             ↑

无论选哪个操作系统（Docker / Debian / Windows），token 值完全相同。选 Docker 只是让页面显示格式与我们的部署方式一致。

此时不要点 Next，也不要执行任何命令。页面下方显示 "No connectors installed" 是正常的，等 Step 10 Docker 部署完成后这里会自动出现 connector。

MAC Step 4：把 Token 存入 AWS Secrets Manager
目的：将 CF tunnel token 加密存储到你自己的 AWS 账号。CMI 没有你的 AWS 账号权限，无法访问。这一步在 Mac 上执行，不在宿主机上。
如果 Secret 名称已存在但处于"待删除"状态：AWS 默认会在 7 天后才真正删除 Secret，这段时间内不能创建同名 Secret。遇到此情况，需换一个不重复的名称，并确保后续所有命令中都使用该新名称。

MAC Step 5：创建 IAM Policy（最小读取权限）
目的：创建一个权限极小的 IAM Policy，只允许读取刚才那一个 Secret，其他任何 AWS 资源均不允许操作。
# 把 ap-southeast-1 替换为你的区域，123456789012 替换为你的 Account ID
cat > /tmp/cf-tunnel-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadTunnelToken",
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "arn:aws:secretsmanager:ap-southeast-1:123456789012:secret:prod/cloudflare/warp-reverse-tunnel-token-aws-*"
    }
  ]
}
EOF

aws iam create-policy \
  --policy-name "CloudflareTunnelTokenReadOnly" \
  --description "Read-only access to Cloudflare tunnel token in Secrets Manager" \
  --policy-document file:///tmp/cf-tunnel-policy.json

需要替换的值

占位符替换为来源
ap-southeast-1你的 AWS 区域代码Step 2 记录的 AWS_REGION
123456789012你的 AWS Account ID（12位数字）Step 2 记录的 AWS_ACCOUNT_ID

Resource 末尾的 -* 是通配符，用于匹配 AWS 自动添加的随机后缀（如 -AbCdEf）。

预期输出： 
{
    "Policy": {
        "PolicyName": "CloudflareTunnelTokenReadOnly",
        "Arn": "arn:aws:iam::123456789012:policy/CloudflareTunnelTokenReadOnly",
        ...
    }
}
记录 Policy ARN（arn:aws:iam::123456789012:policy/CloudflareTunnelTokenReadOnly），下一步用。
MAC Step 6：创建 IAM Role 并绑定 Policy
目的：IAM Role 是可以被 EC2 服务扮演的身份。EC2 实例扮演此 Role 后，自动获得读取 Secret 的临时凭证，无需在宿主机上存储任何 AWS Access Key。
# 第一步：创建 Trust Policy（只允许 EC2 服务扮演此 Role）
cat > /tmp/ec2-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

# 第二步：创建 Role
aws iam create-role \
  --role-name "cf-tunnel-ec2-role" \
  --description "EC2 role for cloudflared, read tunnel token from Secrets Manager" \
  --assume-role-policy-document file:///tmp/ec2-trust-policy.json

# 第三步：绑定 Policy（把 123456789012 替换为你的 Account ID）
aws iam attach-role-policy \
  --role-name "cf-tunnel-ec2-role" \
  --policy-arn "arn:aws:iam::123456789012:policy/CloudflareTunnelTokenReadOnly"

# 第四步：创建 Instance Profile
aws iam create-instance-profile \
  --instance-profile-name "cf-tunnel-ec2-profile"

# 第五步：把 Role 加入 Instance Profile
aws iam add-role-to-instance-profile \
  --instance-profile-name "cf-tunnel-ec2-profile" \
  --role-name "cf-tunnel-ec2-role"
验证： 
aws iam list-attached-role-policies --role-name "cf-tunnel-ec2-role"
{
    "AttachedPolicies": [
        {
            "PolicyName": "CloudflareTunnelTokenReadOnly",
            "PolicyArn": "arn:aws:iam::123456789012:policy/CloudflareTunnelTokenReadOnly"
        }
    ]
}

或通过 AWS Console 创建

  1. IAM → Roles → Create role → AWS service → EC2 → Next
  2. 搜索并勾选 CloudflareTunnelTokenReadOnly → Next
  3. Role name 填 cf-tunnel-ec2-role → Create role
AWS Console / MAC Step 7：把 IAM Role 绑定到 EC2 实例
目的：将 Step 6 的 Role 绑定到 CMI 提供的 EC2 实例，绑定后 EC2 上的程序可自动获得临时 AWS 凭证（每小时轮换），无需手动管理密钥。

方式 A：AWS Console（推荐）

  1. AWS Console → EC2 → Instances
  2. 勾选目标实例 → Actions → Security → Modify IAM role
  3. 下拉选择 cf-tunnel-ec2-role → Update IAM role

方式 B：AWS CLI

# 把 i-xxxxxxxxxxxxxxxxx 替换为你的 EC2 Instance ID（AWS Console → EC2 → Instances 中查看）
aws ec2 associate-iam-instance-profile \
  --instance-id i-xxxxxxxxxxxxxxxxx \
  --iam-instance-profile Name="cf-tunnel-ec2-profile" \
  --region ap-southeast-1

在宿主机上验证绑定成功

# 通过 EC2 元数据服务查询（在宿主机上执行）
curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/
预期输出（Role 名字）： 
cf-tunnel-ec2-role
若无输出，等待 1-2 分钟后重试。

宿主机 Step 9：从 AWS 拉取 Token 并写入本地密钥文件
目的：从 AWS 拉取 token 写入宿主机本地文件，供 Docker 容器以 secrets 方式挂载。目录和文件权限严格限制，普通运维人员无法读取。
# 创建密钥目录（仅 root 可进入）
sudo mkdir -p /opt/cloudflared/secrets
sudo chmod 700 /opt/cloudflared/secrets

# 从 AWS 拉取 token，写入文件（替换区域）
aws secretsmanager get-secret-value \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --region ap-southeast-1 \
  --query "SecretString" \
  --output text \
  | sudo tee /opt/cloudflared/secrets/tunnel.token > /dev/null

# 设置文件权限
# 604 = root 读写，group 无权，other 只读
# 注意：cloudflared 容器内以 UID 65532 运行（非 root）
# 需要 other 有读权限（r--），否则容器报 permission denied
sudo chmod 604 /opt/cloudflared/secrets/tunnel.token
sudo chown root:root /opt/cloudflared/secrets/tunnel.token

# 验证（普通用户 sudo 查看，ec2-user 无法直接访问该目录）
sudo stat /opt/cloudflared/secrets/tunnel.token | grep Access
预期输出： 
Access: (0604/-rw----r--)  Uid: (0/root)   Gid: (0/root)
  • rw-（owner=root）：root 可读写
  • ---（group）：组成员无权限
  • r--（other）：容器内 UID 65532 通过 other 权限读取
  • 目录 chmod 700：普通运维账号连 ls 都报 Permission denied，无法进入目录
宿主机 Step 10：创建 docker-compose.yml 并启动容器
目的：通过 Docker Compose 启动 cloudflared 容器。核心设计：使用 --token-file 而非 --token，token 值不出现在 docker inspect 输出或任何配置文件中。
sudo mkdir -p /opt/cloudflared

sudo tee /opt/cloudflared/docker-compose.yml > /dev/null << 'EOF'
services:
  cloudflared:
    image: cloudflare/cloudflared:latest
    container_name: cloudflared-prod
    restart: unless-stopped
    command: tunnel --no-autoupdate run --token-file /run/secrets/tunnel_token
    secrets:
      - tunnel_token
    tmpfs:
      - /tmp
    logging:
      driver: json-file
      options:
        max-size: "20m"
        max-file: "5"

secrets:
  tunnel_token:
    file: /opt/cloudflared/secrets/tunnel.token
EOF
关键配置说明：
  • --token-file /run/secrets/tunnel_token：token 从文件读取，不出现在进程参数或 docker inspect
  • secrets.tunnel_token.file：指向 Step 9 创建的 root-only 密钥文件，通过 Docker secrets 机制挂载进容器
  • tmpfs: /tmp：防止容器写临时文件到宿主机磁盘
  • restart: unless-stopped：宿主机重启后容器自动恢复

注：已移除 read_only: true——Docker secrets 挂载到 /run/secrets/，该路径在 read_only 模式下容器内 UID 65532 无法读取，会报 permission denied。改用 tmpfs 作为替代安全措施。

sudo docker compose up -d

# 查看状态

预期输出：

宿主机 Step 12：安全防护效果验证（含 root 视角透明说明）
目的：从 CMI 运维人员和系统管理员的角度，逐一测试各种可能的 token 读取手段，诚实呈现哪些被阻断、哪些仍然可见。

测试 1：docker inspect（运维人员最常用的手段）

sudo docker inspect cloudflared-prod | grep -iE "token|eyJ"
结果：防住。输出只含文件路径，无 token 值。这是 --token-file 与传统 --token eyJ... 直传的核心区别。

测试 2：查看 docker-compose.yml

cat /opt/cloudflared/docker-compose.yml | grep -iE "eyJ"
结果：防住。配置文件只含文件路径和镜像名，无任何 token 值。

测试 3：普通运维账号读密钥文件

# 以普通运维账号（非 root）执行
ls /opt/cloudflared/secrets/tunnel.token
cat /opt/cloudflared/secrets/tunnel.token
结果：防住。 
ls: cannot access /opt/cloudflared/secrets/tunnel.token: Permission denied
cat: /opt/cloudflared/secrets/tunnel.token: Permission denied
目录 chmod 700，普通账号连目录都进不去。

测试 4：CMI 尝试访问客户的 AWS Secrets Manager

# 使用 CMI 自己的账号凭证（或无凭证）请求客户的 Secret
AWS_ACCESS_KEY_ID="" AWS_SECRET_ACCESS_KEY="" \
aws secretsmanager get-secret-value \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --region ap-southeast-1 \
  --no-sign-request 2>&1 || true
结果：防住。 
An error occurred (AuthFailure): Request must be signed
CMI 没有客户 AWS 账号的 IAM 凭证，无法访问 Secrets Manager，这是本方案最核心的架构边界。

测试 5：root 通过 docker exec 进入容器读取 secret（⚠️ 剩余暴露）

# 需要 root 权限主动执行
sudo docker exec cloudflared-prod cat /run/secrets/tunnel_token
结果：token 明文可见。

这是本方案已知且公开说明的剩余暴露面。原因：Docker secrets 机制将密钥文件以 tmpfs 方式挂载进容器，容器内进程可读；拥有宿主机 root 权限的人可以通过 docker exec 进入容器读取该文件。

同等路径还包括：sudo cat /proc/$(docker inspect --format '{{.State.Pid}}' cloudflared-prod)/root/run/secrets/tunnel_token

这是 Linux 操作系统的基本属性，不是 cloudflared 或本方案的设计缺陷。任何运行中的进程，其凭证在技术上都可以被 root 读取。

本方案的价值在于：将 token 暴露从"任何人随时可见（配置文件）"提升为"需要 root 主动操作才可见（且每次操作均被审计记录）"。CMI 对此的合规承诺见下方章节。

CMI 合规审计与运营承诺

鉴于本方案的技术透明性——拥有宿主机 root 权限的系统管理员在技术上仍可通过主动操作读取运行中容器内的 token——CMI 承诺以下合规机制，作为技术防护的补充约束：

一、最小权限原则

  • 宿主机的日常运维账号（如 ec2-user）不具备 root 权限，无法进入密钥目录或执行 docker exec
  • root / sudo 权限仅限于特定系统管理员岗位，名单严格管控，实行双人审批机制
  • OpenBao（CMI 内部 Secrets Manager）中的 token KV 路径，CMI 运维账号无读取权限，仅宿主机的 AppRole 凭证可访问

二、全程审计日志

  • 宿主机启用 auditd，记录所有 docker exec、ptrace、/proc 访问等敏感操作，日志不可篡改
  • OpenBao 记录所有 Secret 读写事件，包含操作者身份、时间戳、来源 IP
  • AWS CloudTrail 记录 Secrets Manager 的所有 API 调用（客户可在自己 AWS 账号中查看）
  • 审计日志保留期：不少于 90 天

三、客户可审查权

  • 客户可随时向 CMI 申请审计报告，CMI 将提供涉及该客户 token 的访问记录（仅限该客户的记录，不含其他客户信息）
  • 客户可在自己的 AWS 账号 CloudTrail 中独立查看 Secrets Manager 的访问记录，无需依赖 CMI

四、合同层面约束

  • CMI 与客户签署的服务协议中明确禁止 CMI 员工访问客户的 tunnel token 及其衍生凭证
  • 违反此条款视为重大安全事故，适用相应的赔偿和法律责任条款

五、Token Rotate 建议

  • 建议客户定期 rotate token（最少每季度一次）
  • 人员变动（如 CMI 负责该账号的技术人员离职）时，立即执行 token rotate
  • rotate 后，旧 token 在 Cloudflare 侧立即失效，即使此前被读取也无法继续使用

综上，CMI 的安全保障模型与业界主流的托管服务提供商（如 AWS、Azure）的责任共担原则一致： CMI 承诺不主动访问客户凭证，并通过技术手段提高访问门槛、通过审计机制形成威慑，同时保留客户独立验证的能力。

常见问题排查

问题 1：sudo yum install -y docker 报错 "One of the configured repositories failed"

原因：机器上有损坏的第三方 repo（如 ookla speedtest）导致 yum 整体失败。解决方案：

# 只启用 AWS 官方 repo
sudo yum install -y docker \
  --disablerepo="*" \
  --enablerepo="amzn2-core,amzn2extra-docker"

问题 2：容器启动后反复重启，日志显示 "Failed to read token file: permission denied"

原因：cloudflared 容器内以 UID 65532 运行，文件权限 600 仅 root 可读，容器无法读取。

# 确认容器内用户
sudo docker inspect cloudflared-prod | grep '"User"'
# 应输出 "User": "65532:65532"

# 修正文件权限为 604（other 有读权限）
sudo chmod 604 /opt/cloudflared/secrets/tunnel.token

# 重启容器
cd /opt/cloudflared && sudo docker compose down && sudo docker compose up -d

问题 3：容器运行但 tunnel 显示 Unhealthy 或无连接

sudo docker logs cloudflared-prod --tail 50 | grep -E "ERR|WARN|failed"

常见原因：宿主机安全组不允许出站 7844 端口。确保放行以下出站规则：

  • 443/tcp → api.cloudflare.com
  • 7844/tcp + 7844/udp → region1.v2.argotunnel.com 和 region2.v2.argotunnel.com

问题 4：如何 Rotate Token

  1. CF 控制台 Zero Trust → Networks → Tunnels → 选择 tunnel → Edit → Refresh token，复制新 token
  2. MAC 更新 AWS Secrets Manager：
aws secretsmanager update-secret \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --secret-string "新的 eyJ 开头的完整 token" \
  --region ap-southeast-1
  1. 宿主机 重新拉取并重启：
# 重新从 AWS 拉取新 token 写入密钥文件
aws secretsmanager get-secret-value \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --region ap-southeast-1 \
  --query "SecretString" \
  --output text \
  | sudo tee /opt/cloudflared/secrets/tunnel.token > /dev/null

# 重启容器加载新 token
cd /opt/cloudflared && sudo docker compose restart cloudflared

旧 token 在 Cloudflare 侧立即失效。如怀疑有未授权 connector，追加强制断连：

# ACCOUNT_ID 替换为你的 CF Account ID，TUNNEL_ID 替换为 Tunnel UUID，CLOUDFLARE_API_TOKEN 替换为有权限的 API Token
curl "https://api.cloudflare.com/client/v4/accounts/ACCOUNT_ID/cfd_tunnel/TUNNEL_ID/connections" \
  --request DELETE \
  --header "Authorization: Bearer CLOUDFLARE_API_TOKEN"
