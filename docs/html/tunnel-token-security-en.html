Cloudflared Tunnel Token — Secure Deployment Guide

Goal: Store the CF tunnel token in the customer's own AWS Secrets Manager. CMI runs cloudflared inside a Docker container using --token-file, so the token never appears in plaintext in any config file, process argument, or docker inspect output.

Architecture & Security Boundary

Why this approach is needed

Traditional deployment (--token eyJ... written directly into config) exposes the token statically:

  • docker inspect <container> → token appears in plaintext in the Args field
  • cat /etc/systemd/system/cloudflared.service → token visible in plaintext
  • ps aux → token visible in process arguments

Two-layer isolation

  • Storage isolation: Token is stored in the customer's own AWS Secrets Manager. CMI has no access to the customer's AWS account.
  • Runtime isolation: cloudflared runs in a Docker container using --token-file. The token value never appears in docker inspect, ps aux, or any config file.

│  cloudflared container (runs as UID 65532)                          │

└─────────────────────────────┬────────────────────────────────────────┘

                     Cloudflare Global Network
Transparent disclosure: residual exposure
Any secret held by a running process can technically be read from memory by the host root user. This applies to cloudflared as well:
sudo docker exec cloudflared-prod cat /run/secrets/tunnel_token → token plaintext visible to root

This is a fundamental Linux OS property — not a defect in this solution. This guide addresses static storage exposure (config files, docker inspect, ps aux), which represents the realistic everyday attack surface. The root-level runtime risk is mitigated through CMI's compliance and audit controls — see the Compliance section.

MAC Step 4: Store Token in AWS Secrets Manager
Purpose: Encrypt and store the CF tunnel token in your own AWS account. CMI has no access to your AWS account and cannot retrieve this secret. Run this on your Mac, not on the host.
If a secret with the same name is in "pending deletion" state: AWS holds deleted secrets for 7 days before permanent removal. Use a different name and update all subsequent commands accordingly.

Option A: AWS CLI

aws secretsmanager create-secret \
  --name "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --description "Cloudflare tunnel token for Warp Reverse solution" \
  --secret-string "eyJhIjoiMDI5N2Rm...(your full token from Step 3)" \
  --region ap-southeast-1
Expected output: 
{
    "ARN": "arn:aws:secretsmanager:ap-southeast-1:123456789012:secret:prod/cloudflare/warp-reverse-tunnel-token-aws-AbCdEf",
    "Name": "prod/cloudflare/warp-reverse-tunnel-token-aws"
}
Record the full ARN (including the random suffix like -AbCdEf) — needed in Step 5.
If the output contains "DeletedDate", the secret is pending deletion. Restore it first:
aws secretsmanager restore-secret --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" --region ap-southeast-1

Option B: AWS Console

  1. AWS Console → Secrets Manager → Store a new secret
  2. Secret type: Other type of secret → switch to Plaintext → paste token → Next
  3. Name: prod/cloudflare/warp-reverse-tunnel-token-aws → Next → Next → Store
  4. Open the secret and copy the Secret ARN
MAC Step 5: Create IAM Policy (minimal permissions)
Purpose: Create an IAM Policy that allows reading only this one specific Secret — nothing else in your AWS account.
# Replace ap-southeast-1 with your region, 123456789012 with your Account ID
cat > /tmp/cf-tunnel-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadTunnelToken",
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "arn:aws:secretsmanager:ap-southeast-1:123456789012:secret:prod/cloudflare/warp-reverse-tunnel-token-aws-*"
    }
  ]
}
EOF

aws iam create-policy \
  --policy-name "CloudflareTunnelTokenReadOnly" \
  --description "Read-only access to Cloudflare tunnel token in Secrets Manager" \
  --policy-document file:///tmp/cf-tunnel-policy.json

MAC Step 6: Create IAM Role & Attach Policy
Purpose: An IAM Role is an identity that EC2 can assume. Once the EC2 instance assumes this role, it automatically receives temporary credentials to read the secret — no Access Key file stored anywhere on the host.
# Create Trust Policy (only EC2 service can assume this role)
cat > /tmp/ec2-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

aws iam create-role \
  --role-name "cf-tunnel-ec2-role" \
  --description "EC2 role for cloudflared — read tunnel token from Secrets Manager" \
  --assume-role-policy-document file:///tmp/ec2-trust-policy.json

# Attach Policy (replace 123456789012 with your Account ID)
aws iam attach-role-policy \
  --role-name "cf-tunnel-ec2-role" \
  --policy-arn "arn:aws:iam::123456789012:policy/CloudflareTunnelTokenReadOnly"

aws iam create-instance-profile \
  --instance-profile-name "cf-tunnel-ec2-profile"

aws iam add-role-to-instance-profile \
  --instance-profile-name "cf-tunnel-ec2-profile" \
  --role-name "cf-tunnel-ec2-role"
Verify: 
aws iam list-attached-role-policies --role-name "cf-tunnel-ec2-role"
# Expected: CloudflareTunnelTokenReadOnly listed

HOST Step 9: Pull Token from AWS & Write to Secret File
Purpose: Pull the token from AWS and write it to a local file on the host, which Docker will mount into the container as a secret. Directory and file permissions are strictly locked down so regular ops accounts cannot read the file.
# Create secret directory (root-only entry)
sudo mkdir -p /opt/cloudflared/secrets
sudo chmod 700 /opt/cloudflared/secrets

# Pull token from AWS and write to file
aws secretsmanager get-secret-value \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --region ap-southeast-1 \
  --query "SecretString" \
  --output text \
  | sudo tee /opt/cloudflared/secrets/tunnel.token > /dev/null

# Set permissions: 604 = root rw, group none, other r
# IMPORTANT: cloudflared container runs as UID 65532 (non-root)
# The container needs "other" read (r--) — chmod 600 causes permission denied
sudo chmod 604 /opt/cloudflared/secrets/tunnel.token
sudo chown root:root /opt/cloudflared/secrets/tunnel.token

# Verify (regular users cannot even ls the directory)
sudo stat /opt/cloudflared/secrets/tunnel.token | grep Access
Expected output: 
Access: (0604/-rw----r--)  Uid: (0/root)   Gid: (0/root)
  • rw- (owner=root): root can read/write
  • --- (group): no group access
  • r-- (other): container UID 65532 reads via "other" permission
  • Directory chmod 700: regular ops accounts get Permission denied even on ls
HOST Step 10: Create docker-compose.yml & Start Container
Purpose: Start the cloudflared container via Docker Compose. The key design: --token-file instead of --token, so the token value never appears in docker inspect or any config file.
sudo mkdir -p /opt/cloudflared

sudo tee /opt/cloudflared/docker-compose.yml > /dev/null << 'EOF'
services:
  cloudflared:
    image: cloudflare/cloudflared:latest
    container_name: cloudflared-prod
    restart: unless-stopped
    command: tunnel --no-autoupdate run --token-file /run/secrets/tunnel_token
    secrets:
      - tunnel_token
    tmpfs:
      - /tmp
    logging:
      driver: json-file
      options:
        max-size: "20m"
        max-file: "5"

secrets:
  tunnel_token:
    file: /opt/cloudflared/secrets/tunnel.token
EOF
Key design decisions:
SettingReason
--token-file not --tokenToken value never appears in docker inspect or process args
secrets.tunnel_token.fileDocker secrets mechanism mounts the file into the container
tmpfs: /tmpPrevents container from writing temp files to host disk
No read_only: trueDocker secrets mount to /run/secrets/; in read_only mode UID 65532 cannot read it → permission denied. tmpfs provides equivalent protection.
cd /opt/cloudflared
sudo docker compose up -d
sudo docker compose ps
Expected: 
NAME               IMAGE                           STATUS
cloudflared-prod   cloudflare/cloudflared:latest   Up X seconds

HOST Step 12: Security Verification (incl. root-level transparency)
Purpose: Test each possible token-reading approach from a CMI operator's perspective, honestly showing what is blocked and what remains accessible.

Test 1: docker inspect

sudo docker inspect cloudflared-prod | grep -iE "token|eyJ"
Blocked. Output contains only file paths — no token value.

Test 2: Read docker-compose.yml

cat /opt/cloudflared/docker-compose.yml | grep -iE "eyJ"
Blocked. Config file contains no token value.

Test 3: Regular ops account reads secret file

ls /opt/cloudflared/secrets/tunnel.token
cat /opt/cloudflared/secrets/tunnel.token
Blocked. 
ls: cannot access /opt/cloudflared/secrets/tunnel.token: Permission denied
cat: /opt/cloudflared/secrets/tunnel.token: Permission denied
Directory chmod 700 — non-root accounts cannot even enter the directory.

Test 4: CMI attempts to access customer AWS Secrets Manager

AWS_ACCESS_KEY_ID="" AWS_SECRET_ACCESS_KEY="" \
aws secretsmanager get-secret-value \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --region ap-southeast-1 \
  --no-sign-request 2>&1 || true
Blocked. 
An error occurred (AuthFailure): Request must be signed
CMI has no customer AWS credentials — this is the core architectural boundary of this solution.

Test 5: Root reads token via docker exec — ⚠️ RESIDUAL EXPOSURE

sudo docker exec cloudflared-prod cat /run/secrets/tunnel_token
Token plaintext is visible to root.

This is an acknowledged and intentionally disclosed residual exposure. Any secret held by a running process is technically readable from memory by the host root user — this is a fundamental Linux OS property, not a flaw in this design.

Equivalent approach: sudo cat /proc/$(docker inspect --format '{{.State.Pid}}' cloudflared-prod)/root/run/secrets/tunnel_token

What this solution does protect against: static storage exposure via config files, docker inspect, ps aux — the realistic everyday attack vectors.

What mitigates the root-level risk: CMI's compliance controls — see below.

CMI Compliance & Audit Commitments

Given the technical transparency above — a host root user can technically read the token from a running container — CMI commits to the following controls as a compliance layer over the technical safeguards:

Least Privilege

  • Daily ops accounts (e.g., ec2-user) have no root or sudo access — they cannot enter the secret directory or run docker exec
  • Root / sudo is restricted to named sysadmins with dual-approval workflow
  • CMI ops accounts have no read permission on any customer token path in OpenBao (CMI's internal secret manager)

Full Audit Logging

  • auditd on host records all docker exec, ptrace, /proc access — logs are tamper-resistant
  • OpenBao logs every secret read/write event with operator identity, timestamp, and source IP
  • AWS CloudTrail logs all Secrets Manager API calls — customers can view independently in their own AWS account
  • Log retention: minimum 90 days

Customer Audit Rights

  • Customers may request filtered audit reports at any time — CMI provides only that customer's events, never other customers' data
  • Customers can verify independently via their own AWS CloudTrail without relying on CMI

Contractual Constraint

  • CMI's service agreement explicitly prohibits accessing customer tunnel tokens
  • Violations constitute a critical security incident with defined liability provisions

Token Rotation Recommendations

  • Rotate at minimum quarterly
  • Rotate immediately upon any relevant personnel change on CMI's side
  • After rotation, the old token is immediately invalidated on Cloudflare's side — even if previously read, it cannot be reused

CMI's security model is consistent with industry-standard managed service frameworks (analogous to the AWS Shared Responsibility Model): CMI commits not to access customer credentials proactively, raises the technical barrier to access, establishes a deterrent through audit, and preserves the customer's right to independently verify.

Troubleshooting

Issue 1: yum install docker fails — "One of the configured repositories failed"

A broken third-party repo (e.g., ookla speedtest) blocks yum entirely. Use AWS official repos only:

sudo yum install -y docker \
  --disablerepo="*" \
  --enablerepo="amzn2-core,amzn2extra-docker"

Issue 2: Container keeps restarting — "Failed to read token file: permission denied"

# Confirm container user
sudo docker inspect cloudflared-prod | grep '"User"'
# Should show: "User": "65532:65532"

# Fix permissions (604 not 600)
sudo chmod 604 /opt/cloudflared/secrets/tunnel.token

# Restart
cd /opt/cloudflared && sudo docker compose down && sudo docker compose up -d

Issue 3: Tunnel Unhealthy / no connections

sudo docker logs cloudflared-prod --tail 50 | grep -E "ERR|WARN|failed"

Common cause: outbound ports blocked. Ensure these outbound rules are open:

  • 443/tcp → api.cloudflare.com
  • 7844/tcp + 7844/udp → region1.v2.argotunnel.com and region2.v2.argotunnel.com

Issue 4: How to rotate the token

BROWSER CF Dashboard → Networks → Tunnels → tunnel → Edit → Refresh token → copy new token

MAC Update AWS Secrets Manager:

aws secretsmanager update-secret \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --secret-string "eyJ...(new full token)" \
  --region ap-southeast-1

HOST Re-pull and restart:

aws secretsmanager get-secret-value \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --region ap-southeast-1 \
  --query "SecretString" --output text \
  | sudo tee /opt/cloudflared/secrets/tunnel.token > /dev/null

cd /opt/cloudflared && sudo docker compose restart cloudflared

If an unauthorized connector may have connected, force-disconnect all:

# Replace ACCOUNT_ID, TUNNEL_ID, CLOUDFLARE_API_TOKEN with real values
curl "https://api.cloudflare.com/client/v4/accounts/ACCOUNT_ID/cfd_tunnel/TUNNEL_ID/connections" \
  --request DELETE \
  --header "Authorization: Bearer CLOUDFLARE_API_TOKEN"
