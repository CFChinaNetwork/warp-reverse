整体架构说明

在开始操作之前，先理解整体的安全设计思路。

核心问题

传统方式把 CF tunnel token 直接写进 systemd service 文件或 docker run 命令参数，CMI 用 cat 或 docker inspect 就能看到明文 token。

解决思路

两层隔离：① token 存储在客户自己的 AWS 账号，CMI 无法访问；② cloudflared 跑在 Docker 容器里，用 --token-file 读取密钥文件，token 值不出现在任何命令行参数或 docker inspect 输出中。

┌──────────────────────────────────────────────────────────────────┐
│              客户自己的 AWS 账号（CMI 无权访问）                    │
│   AWS Secrets Manager                                             │
│   └── prod/cloudflare/warp-reverse-tunnel-token-aws  ← token 加密存储             │
│         只允许绑定到宿主机的 IAM Role 读取                         │
└──────────────────────────┬───────────────────────────────────────┘
                           │ Instance Profile 自动认证（无需密钥文件）
                           │ CMI 没有客户 AWS 凭证 → 403 拒绝
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│                    CMI 提供的宿主机                                │
│                                                                   │
│   1. AWS CLI 拉取 token → 写入 /opt/cloudflared/secrets/         │
│      （chmod 600，root-only，运维人员无法读取）                    │
│                                                                   │
│   2. docker-compose.yml                                           │
│      secrets.tunnel_token.file → 上面的密钥文件                  │
│      command: tunnel run --token-file /run/secrets/tunnel_token  │
│      read_only: true                                             │
│                                                                   │
│   CMI 运维人员能看到什么？                                         │
│   ✓ docker-compose.yml（只含文件路径，无 token 值）               │
│   ✗ docker inspect 不暴露 token（--token-file 方式）              │
│   ✗ AWS Secrets Manager（无 IAM 权限）                           │
└──────────────────────────┬───────────────────────────────────────┘
                           │ QUIC / TLS 1.3
                           ▼
                   Cloudflare 全球网络

MAC Step 1：在 Mac 上配置 AWS CLI 认证
目的：本步骤让你的 Mac 能通过命令行操作 AWS，包括后续创建 Secret、IAM Policy 等。如果你更习惯在 AWS Console（网页）操作，本步可以跳过，所有操作都在网页完成。

aws --version

浏览器 Step 3：在 CF Dashboard 创建 Tunnel 并获取 Token
目的：在 Cloudflare 控制台创建一个 tunnel，拿到对应的 token 字符串（以 eyJ 开头）。注意：这一步只需要创建 tunnel 并复制 token，不在 VM 上执行任何 CF 给出的安装命令，我们用后面的安全方式部署。
  1. 打开 https://one.dash.cloudflare.com，选择你的账号
  2. 左侧菜单：Networks → Connectors → Cloudflare Tunnels
  3. 点击 Create a tunnel
  4. Connector type 选择 Cloudflared，点击 Next
  5. Tunnel name 填写一个有意义的名字，例：prod-server-tunnel
  6. 点击 Save tunnel
  7. 下一页 "Install and run a connector"，Operating system 选择 Docker
  8. 页面显示安装命令，格式如下：
页面显示的命令长这样：
docker run cloudflare/cloudflared:latest tunnel --no-autoupdate run --token eyJhIjoiMDI5N2RmMzE5OWE5NGM4MDgyZDA5Ym...
                                                                             ↑
                                                                     这一长串就是 token，以 eyJ 开头
只复制 token 本身（eyJ 开头的那一长串字符），不要复制整个命令。
token 通常有几百个字符长，一直复制到末尾。

注意：无论在这里选择哪个操作系统（Docker / Debian / Windows 等），token 值都是完全相同的。选 Docker 只是为了让页面显示的命令格式与我们的部署方式保持一致，方便理解。我们后续不会直接使用页面给出的这条命令，而是用 --token-file 方式替代 --token 直接传值。

记录以下信息

变量说明你的实际值
CF_TUNNEL_TOKENeyJ 开头的完整 token 字符串eyJhIjoiMDI5N2Rm...
CF_TUNNEL_NAME你给 tunnel 起的名字prod-server-tunnel
此时不要点 Next，也不要执行页面给出的 docker run 命令。
复制完 token 后，保留这个页面即可（或直接关闭）。
页面下方 Connectors 区域会显示 "No connectors installed"，这是正常的—— tunnel 已创建，但还没有 connector 接入。 等我们后续完成 Step 8～10 的 Docker 部署后，这里会自动出现已连接的 connector。
MAC Step 4：把 Token 存入 AWS Secrets Manager
目的：这是整个方案的核心。把 CF tunnel token 加密存储在你自己的 AWS 账号里。CMI 没有你的 AWS 账号权限，永远无法访问这个 Secret。这一步在你自己的 Mac 上执行，不在 VM 上。

方式 A：通过 AWS CLI 命令创建（在 Mac 终端执行）

# 注意：把下面两处红色标注替换成你的真实值
aws secretsmanager create-secret \
  --name "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --description "Cloudflare tunnel token for production server" \
  --secret-string "eyJhIjoiMDI5N2RmMzE5OWE5NGM4MDgyZDA5...(替换成你在Step3复制的完整token)" \
  --region ap-southeast-1

需要替换的值

占位符替换为来源
eyJhIjoiMDI5N...Step 3 复制的完整 CF tunnel tokenCF Dashboard
ap-southeast-1你的 AWS 区域代码Step 2 记录的 AWS_REGION

MAC Step 5：创建 IAM Policy（最小权限读取 Secret）
目的：IAM Policy 是 AWS 的权限规则，规定"谁能做什么操作、对哪些资源"。这一步创建一个权限极小的 Policy，只允许读取刚才那一个 Secret，其他任何 AWS 资源都不允许操作。这样即使这个权限被滥用，损失也被控制到最小。

第一步：把 Policy 内容保存为本地 JSON 文件

在你的 Mac 终端执行（注意替换两处红色标注）：

# 创建 Policy JSON 文件
# 把 ap-southeast-1 替换成你的区域
# 把 123456789012 替换成你的 AWS Account ID（Step 2 记录的12位数字）

cat > /tmp/cf-tunnel-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadTunnelToken",
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "arn:aws:secretsmanager:ap-southeast-1:123456789012:secret:prod/cloudflare/warp-reverse-tunnel-token-aws-*"
    }
  ]
}
EOF

需要替换的值（重要！）

占位符替换为来源示例
ap-southeast-1 你创建 Secret 时使用的 AWS 区域代码 Step 2 记录的 AWS_REGION ap-southeast-1 / ap-east-1
123456789012 你的 AWS 账号 ID（12位纯数字） Step 2 记录的 AWS_ACCOUNT_ID 你的真实12位数字

为什么 Resource 末尾是 -*？
AWS 在 Secret 名称后面会自动添加随机后缀（如 -AbCdEf）。用通配符 * 确保能匹配到带任意后缀的 Secret ARN。

第二步：用 CLI 创建 Policy

aws iam create-policy \
  --policy-name "CloudflareTunnelTokenReadOnly" \
  --description "Allows read-only access to Cloudflare tunnel token secret" \
  --policy-document file:///tmp/cf-tunnel-policy.json
预期输出： 
{
    "Policy": {
        "PolicyName": "CloudflareTunnelTokenReadOnly",
        "PolicyId": "ANPAI...",
        "Arn": "arn:aws:iam::123456789012:policy/CloudflareTunnelTokenReadOnly",
        "CreateDate": "...",
        "AttachmentCount": 0,
        "IsAttachable": true
    }
}

记录以下信息

变量说明你的实际值
POLICY_ARN输出里的 Arn 字段（arn:aws:iam:: 开头）arn:aws:iam::123456789012:policy/CloudflareTunnelTokenReadOnly

  3. Next → Policy name 填 CloudflareTunnelTokenReadOnly → Create policy
MAC Step 6：创建 IAM Role 并绑定 Policy
目的：IAM Role 是一个可以被 AWS 服务"扮演"的身份。这里创建一个专门给 EC2 使用的 Role，并把 Step 5 的 Policy 附加上去。之后 EC2 实例扮演这个 Role，就自动获得读取 Secret 的权限，无需在 VM 上存储任何 AWS 密钥。
# 第一步：创建 Trust Policy（规定只有 EC2 服务可以扮演此 Role）
cat > /tmp/ec2-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

# 第二步：创建 Role
aws iam create-role \
  --role-name "cf-tunnel-ec2-role" \
  --description "Role for EC2 running cloudflared, allows reading tunnel token from Secrets Manager" \
  --assume-role-policy-document file:///tmp/ec2-trust-policy.json

# 第三步：把 Policy 附加到 Role
# 把 123456789012 替换成你的 AWS Account ID
aws iam attach-role-policy \
  --role-name "cf-tunnel-ec2-role" \
  --policy-arn "arn:aws:iam::123456789012:policy/CloudflareTunnelTokenReadOnly"

# 第四步：创建 Instance Profile（EC2 通过这个来使用 Role）
aws iam create-instance-profile \
  --instance-profile-name "cf-tunnel-ec2-profile"

# 第五步：把 Role 加入 Instance Profile
aws iam add-role-to-instance-profile \
  --instance-profile-name "cf-tunnel-ec2-profile" \
  --role-name "cf-tunnel-ec2-role"
验证 Role 创建成功： 
aws iam get-role --role-name "cf-tunnel-ec2-role"
{
    "Role": {
        "RoleName": "cf-tunnel-ec2-role",
        "Arn": "arn:aws:iam::123456789012:role/cf-tunnel-ec2-role",
        ...
    }
}

或通过 AWS Console 网页创建 Role

  1. AWS Console → IAM → Roles → Create role
  2. Trusted entity type 选 AWS service，Use case 选 EC2 → Next
  3. 搜索框输入 CloudflareTunnelTokenReadOnly，勾选该 Policy → Next
  4. Role name 填 cf-tunnel-ec2-role → Create role
浏览器 / MAC Step 7：把 IAM Role 绑定到 EC2 实例
目的：把 Step 6 创建的 Role 绑定到 CMI 提供的 EC2 实例上。绑定后，EC2 内部的程序可以自动获得临时 AWS 凭证（每小时自动轮换），完全不需要在 VM 上手动配置任何 AWS Key。
你需要先知道 EC2 实例的 Instance ID。
格式：i-xxxxxxxxxxxxxxxxx（i- 开头后接17位十六进制字符）
查找方式：AWS Console → EC2 → Instances → 找到目标实例 → 复制 Instance ID 列的值

方式 A：通过 AWS Console 网页操作（直观）

  1. AWS Console → EC2 → Instances
  2. 勾选你的 EC2 实例
  3. 点击 Actions → Security → Modify IAM role
  4. IAM role 下拉选择：cf-tunnel-ec2-role
  5. 点击 Update IAM role

方式 B：通过 AWS CLI 命令操作

# 把 i-xxxxxxxxxxxxxxxxx 替换成你的实际 EC2 Instance ID
# 把 ap-southeast-1 替换成你的区域
aws ec2 associate-iam-instance-profile \
  --instance-id i-xxxxxxxxxxxxxxxxx \
  --iam-instance-profile Name="cf-tunnel-ec2-profile" \
  --region ap-southeast-1

在 VM 上验证 Role 绑定成功

SSH 登录到 EC2，执行以下命令：

# 查询实例当前绑定的 IAM Role（利用 EC2 元数据服务查询）
curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/
预期输出（就是 Role 的名字）： 
cf-tunnel-ec2-role
如果没有输出或返回 404，等待 1-2 分钟后重试（绑定需要短暂生效时间）。 
# 进一步验证：查看临时凭证（由 AWS 自动生成，每小时自动轮换）
curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/cf-tunnel-ec2-role
预期输出： 
{
  "Code": "Success",
  "Type": "AWS-HMAC",
  "AccessKeyId": "ASIA...",     ← 临时 Key，由 AWS 自动管理
  "SecretAccessKey": "...",
  "Token": "AQoD...",
  "Expiration": "2026-04-08T16:00:00Z"   ← 约1小时后到期并自动续期
}
能看到 "Code": "Success" 说明 Instance Profile 工作正常。
宿主机 Step 8：安装 Docker 和 AWS CLI
目的：宿主机上需要两个工具：Docker 用来运行 cloudflared 容器；AWS CLI 用来从 Secrets Manager 拉取 token 并写入本地密钥文件。cloudflared 本身不直接安装在宿主机上，完全在容器内运行。

SSH 登录到宿主机，执行以下命令：

安装 Docker

根据你的系统选择对应的安装方式：

Amazon Linux 2（本手册使用的环境）

注意：Amazon Linux 2 不支持通用的 get.docker.com 脚本，必须使用 AWS 官方的 amzn2extra-docker repo 安装。如果机器上安装了第三方 repo（如 ookla speedtest），需要禁用后再安装，否则 yum 会因 repo 元数据损坏而整体失败。
# 只使用 AWS 官方 repo 安装（绕过第三方损坏的 repo）
sudo yum install -y docker \
  --disablerepo="*" \
  --enablerepo="amzn2-core,amzn2extra-docker"

# 启动并设置开机自启
sudo systemctl start docker
sudo systemctl enable docker

# 让当前用户无需 sudo 运行 docker

宿主机 Step 9：从 AWS 拉取 Token 并写入本地密钥文件
目的：从 AWS Secrets Manager 拉取 token，写入宿主机本地的 root-only 密钥文件，供 Docker 容器以 secret 方式挂载。这个文件权限设为 600（仅 root 可读），CMI 普通运维人员无法直接读取。后续 token rotate 时只需重复这一步并重启容器即可。
# 创建密钥存储目录（仅 root 可进入）
sudo mkdir -p /opt/cloudflared/secrets
sudo chmod 700 /opt/cloudflared/secrets

# 从 AWS Secrets Manager 拉取 token，写入密钥文件
# 把 ap-southeast-1 替换成你的 AWS 区域
aws secretsmanager get-secret-value \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --region ap-southeast-1 \
  --query "SecretString" \
  --output text \
  | sudo tee /opt/cloudflared/secrets/tunnel.token > /dev/null

# 设置权限：root 可读写，其他用户只读（604）
# 注意：cloudflared 容器内以 UID 65532 运行（非 root），
# 需要 other 有读权限（604），否则容器报 permission denied
sudo chmod 604 /opt/cloudflared/secrets/tunnel.token
sudo chown root:root /opt/cloudflared/secrets/tunnel.token

# 验证文件权限（用 sudo，ec2-user 无法访问该目录）
sudo stat /opt/cloudflared/secrets/tunnel.token | grep Access
预期输出： 
Access: (0604/-rw----r--)  Uid: (0/root)   Gid: (0/root)
  • root 可读写（rw-）
  • group 无权限（---）
  • other 只读（r--）—— 容器内 UID 65532 通过 other 权限读取
  • CMI 普通运维账号无法进入 /opt/cloudflared/secrets/ 目录（目录权限 700），仍然无法直接读取文件

宿主机 Step 10：创建 docker-compose.yml 并启动容器
目的：通过 Docker Compose 启动 cloudflared 容器。关键设计：使用 Docker secrets 机制将密钥文件挂载进容器，配合 --token-file 参数读取，token 值不出现在 docker inspect 输出或任何配置文件的明文位置。
# 创建工作目录
sudo mkdir -p /opt/cloudflared
sudo tee /opt/cloudflared/docker-compose.yml > /dev/null << 'EOF'
services:
  cloudflared:
    image: cloudflare/cloudflared:latest
    container_name: cloudflared-prod
    restart: unless-stopped
    read_only: true
    command: tunnel --no-autoupdate run --token-file /run/secrets/tunnel_token
    secrets:
      - tunnel_token
    logging:
      driver: json-file
      options:
        max-size: "20m"
        max-file: "5"

secrets:
  tunnel_token:
    file: /opt/cloudflared/secrets/tunnel.token
EOF
配置说明：
  • --token-file /run/secrets/tunnel_token — token 从文件读取，不出现在进程参数或 docker inspect 中
  • read_only: true — 容器根文件系统只读，防止容器内程序写文件
  • secrets.tunnel_token.file — 指向 Step 9 创建的 root-only 密钥文件
  • restart: unless-stopped — 宿主机重启后容器自动启动
# 启动容器
cd /opt/cloudflared
sudo docker compose up -d

# 查看容器状态
sudo docker compose ps
预期输出： 
NAME                IMAGE                          COMMAND                  SERVICE        STATUS
cloudflared-prod    cloudflare/cloudflared:latest  "/bin/cloudflared tu…"   cloudflared    Up 30 seconds

常见问题排查

问题 1：容器启动失败，日志显示 AccessDeniedException

# 检查 IAM Role 是否已绑定到宿主机
curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/
# 应输出：cf-tunnel-ec2-role

# 手动测试能否拉取 Secret
aws secretsmanager get-secret-value \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --region ap-southeast-1 \
  --query SecretString \
  --output text

如果报错，检查 Step 5 的 IAM Policy 里 Resource 的区域和 Account ID 是否填写正确。

问题 2：容器运行但 tunnel 显示 Unhealthy

# 查看容器详细日志
sudo docker logs cloudflared-prod --tail 50 | grep -E "ERR|WARN|failed"

常见原因：宿主机安全组不允许出站 443 或 7844 端口。确保放行 443/tcp 和 7844/tcp（TCP+UDP）出站至 region1.v2.argotunnel.com 和 region2.v2.argotunnel.com。

问题 3：如何 Rotate Token（发现疑似泄露时）

  1. 浏览器 登录 Cloudflare One → Networks → Tunnels → 选择 tunnel → Edit → Refresh token，复制新 token
  2. MAC 更新 AWS Secrets Manager 中的 token：
aws secretsmanager update-secret \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --secret-string "新的 eyJ 开头的 token" \
  --region ap-southeast-1
  1. 宿主机 重新从 AWS 拉取新 token 并写入密钥文件，然后重启容器：
# 重新拉取新 token 写入密钥文件
aws secretsmanager get-secret-value \
  --secret-id "prod/cloudflare/warp-reverse-tunnel-token-aws" \
  --region ap-southeast-1 \
  --query "SecretString" \
  --output text \
  | sudo tee /opt/cloudflared/secrets/tunnel.token > /dev/null

# 重启容器，加载新 token
cd /opt/cloudflared && sudo docker compose restart cloudflared

旧 token 在 Cloudflare 侧立即失效。如怀疑已有未授权 connector 接入，还需通过 API 强制断开所有连接：

curl "https://api.cloudflare.com/client/v4/accounts/$ACCOUNT_ID/cfd_tunnel/$TUNNEL_ID/connections" \
  --request DELETE \
  --header "Authorization: Bearer $CLOUDFLARE_API_TOKEN"
